# 前端工作流和验证规则

## 任务分类

开始修改前，先明确任务属于哪一类：

- 缺陷修复：先复现，再定位根因，最后添加回归测试。
- 功能开发：先核对需求和后端契约，再设计数据流和页面状态。
- 接口变更：同时检查后端 DTO、响应包装、权限、前端类型、调用方和测试。
- 重构：先锁定已有行为，禁止借重构名义改变权限、字段或错误语义。
- 安全修复：优先确认信任边界、攻击路径、影响范围和相邻入口。
- 文档或规则：以当前源码为依据，不能把历史文档当作接口事实。

## 修改前检查

1. 读取仓库 AGENTS.md、frontend/AGENTS.md 和本目录四类核心文件。
2. 检查 git status，确认是否存在用户修改；不得覆盖或回滚无关改动。
3. 使用 rg 查找目标组件、调用方、接口路径、字段名、权限编码和相关测试。
4. 阅读目标文件及其同级实现，不根据文件名猜测架构。
5. 需要后端契约时，阅读 service/controller、service/entity/dto、service/config 和对应测试。
6. 写出最小契约表：

~~~text
输入：字段、类型、来源、校验、是否敏感
输出：响应结构、空值、分页、二进制或流
错误：HTTP 状态、业务 code、用户提示、重试策略
权限：登录要求、权限编码、租户和数据范围
状态：加载、空数据、提交中、成功、失败、取消
副作用：缓存、路由、文件、对象 URL、定时器、存储
验证：类型检查、单测、构建、集成检查
~~~

## 实现顺序

1. 先定义或确认边界类型、请求类型、响应类型和运行时校验。
2. 在 API 模块中实现接口，不在页面中散落 URL、请求头和响应解包。
3. 在 Store 或组合式函数中编排状态、刷新、取消和错误处理。
4. 页面和组件只负责展示、交互和状态呈现。
5. 处理成功、空数据、校验失败、401、403、限流、网络错误和取消。
6. 涉及动态路由时，先验证服务端数据，再匹配本地组件白名单，最后注册路由。
7. 涉及文件时，明确 JSON 与二进制模式，释放对象 URL，处理文件名和失败响应。
8. 只在完成行为和测试后进行局部格式化，不格式化无关目录。

## 缺陷修复要求

- 必须说明现象、复现路径、根因和最小修复范围。
- 修复应优先覆盖用户给出的真实错误、请求、字段或页面，不先做大范围重构。
- 回归测试必须证明原缺陷路径已恢复，并覆盖最接近的错误或边界条件。
- 如果无法复现，继续做静态调用链和契约分析，并明确说明证据不足，不得声称已修复。

## 功能开发要求

- 先确认是否已有同类页面、Store、API 模块和组件可复用。
- 新增接口时必须确认后端路由前缀、HTTP 方法、请求编码、DTO 字段、权限和响应模式。
- 新增写操作时必须确认重复提交、幂等、并发、回滚和失败提示。
- 新增列表时必须确认服务端分页、筛选、排序、租户和字段权限。
- 新增权限按钮时必须使用后端已存在或明确新增的权限编码；前端显示控制不能代替后端依赖。

## 验证命令

前端默认在 frontend/ 目录执行：

~~~sh
pnpm type-check
pnpm test:unit -- --run
pnpm build
pnpm format
~~~

格式化命令只对明确涉及的源文件执行，避免产生无关差异。交付前还要执行：

~~~sh
git diff --check
git status --short
~~~

后端若被明确纳入联动任务，在 service/ 目录执行适用命令：

~~~sh
poetry run python -m pytest -q
poetry run python -m pytest -q -m "not integration" --cov --cov-report=term-missing
poetry run black .
poetry run isort .
poetry run flake8 .
~~~

需要真实 MySQL 或 Redis 的测试必须显式标记为集成验证，并使用随机临时数据和自身清理。Docker、迁移、真实 OSS、SMTP、Webhook、OIDC 或 LDAP 未执行时，不得声称已完成运行时验证。

## 停止并报告

遇到以下情况必须停止扩大范围并向用户说明：

- 需求缺少会改变接口、字段、权限或数据语义的关键选择。
- 现有用户修改与目标文件冲突，无法安全合并。
- 需要删除数据库、卷、生产文件或大量用户数据。
- 需要更改令牌算法、密码策略、租户隔离、管理员保护或跨租户权限。
- 需要真实外部网络、第三方服务或远程部署权限。
- 依赖、Node、Python、Docker 或数据库环境阻止验证。
